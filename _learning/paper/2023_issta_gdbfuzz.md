## 背景

### 一、覆盖率引导的模糊测试

几十年前，Miller 等人用随机数据测试 Unix 命令行工具，观察到由软件漏洞导致的崩溃，并将此方法称为“模糊测试”。从设计上讲，模糊测试只能检测实际执行的代码中的漏洞，这就是为什么达到高代码覆盖率是可取的。如今，已经开发了海量不同的模糊测试技术，主要分为基于模型的模糊测试（根据输入语言的规范生成测试数据）和基于变异的模糊测试（对程序的已知输入进行随机变异）。基于变异的模糊测试很有吸引力，因为只需要目标程序的少量样本输入（种子）。这些种子最初被添加到基础输入集合（语料库）中，从这些集合中生成变异的测试输入。当触发目标程序先前未见过行为的测试输入被添加到语料库时，基于变异的模糊测试特别有效，这称为**反馈驱动的模糊测试**。对于覆盖率引导的模糊测试，执行新的代码路径或块被视为先前未见的行为。

``` c
void process_data ( char * buffer , unsigned int length ) {
    char stack_array [20];
    if( length > 0 && buffer [0] == 'b')
        if( length > 1 && buffer [1] == 'u')
            if( length > 2 && buffer [2] == 'g')
                if( length > 3 && buffer [3] == '!')
                    memcpy ( stack_array , buffer , length );
}
```

考虑上述代码中函数 `process_data` 的代码，当输入的前四个字符匹配 "bug!" 且长度值大于 20 时，会导致栈溢出。虽然黑盒模糊测试器需要一次从 \(2^{8 \times 4} = 2^{32}\) 种组合中正确猜出前四个字符，但覆盖率引导的模糊测试器可以在每个比较步骤上以 \(2^{8}\) 种可能组合单独取得进展，从而增加了在模糊测试期间生成触发栈溢出的输入的整体概率。

典型的用户程序通过利用源代码插桩进行模糊测试，这样覆盖率就可以通过额外插入的代码反馈给模糊测试器。或者，当没有源代码，因此在编译时无法进行代码插桩时，使用模拟器来获取代码覆盖率反馈。

### 二、调试器和断点

调试器是观察程序执行的常用工具，特别是用于理解意外的程序行为。用户程序的一个流行调试器是 GDB。它可以在期望的点暂停目标程序的执行，检查内存值，并单步执行代码。微控制器具有不同类型的调试接口，例如联合测试行动组 (JTAG) 端口。使用这样的端口，调试探针可以直接访问硬件。调试探针可以实现 GDB 远程存根，这样 GDB 就可以通过 GDB 远程串行协议执行调试操作。通常，GDB 运行在托管开发环境和应用程序源文件的程序员 PC 上。根据我们的观察，大多数可用的板载和板外微控制器调试器都实现了 GDB 远程存根，例如来自 Segger、STMicroelectronics 或 Lauterbach 的调试器。因此，它可以作为深入了解嵌入式系统执行的通用方式。

商用设备上的调试接口理想情况下是封闭或禁用的，以防止攻击。然而，过去已经多次表明，被禁用的调试接口可以通过故障注入攻击（如电源或时钟毛刺）重新打开。此外，可以将商用设备的固件传输到具有可访问调试接口的等效开发板，从而启用调试。

调试器使用断点在所需位置停止执行。嵌入式中的两类断点：
- **软件断点**是通过将软件二进制文件中的原始指令替换为一个特殊的指令字来实现的，该指令字在执行时会触发中断。恢复执行时，调试器会重新插入原始指令。因此，软件断点需要重写目标程序的一小部分。
- 持续重写内存可能需要时间，并且会磨损设备（闪存）存储器。如果程序存储在只读存储器上，则这种重写是不可能的。因此，微控制器通常配备有一定数量的**硬件断点**。硬件断点对应于微处理器上的实际寄存器，一旦激活，当程序计数器值等于其寄存器值时，就会中断程序执行。因此，硬件断点可以设置在任何程序地址，无论相应的代码存储在何种类型的内存中。

有一些方法使用软件断点来测量代码覆盖率和获取模糊测试反馈，以避免源代码插桩的开销和障碍，其思想是将软件断点插入未到达的基本块，从而允许程序以全速执行，直到达到新的覆盖率。一旦执行遇到断点，就从二进制文件中移除相应的指令以避免进一步的开销。Oh 等人使用软件断点测量嵌入式固件中的代码覆盖率。他们在编译期间提取程序每个基本块的起始地址，并在每个地址插入软件断点，一旦被命中就移除它们。

### 三、控制流图

由于我们希望利用硬件断点进行模糊测试反馈，我们首先需要确定目标程序驻留在哪些内存地址。简单来说，可以考虑设备可执行内存区域内的所有地址。然而，只有一小部分内存地址包含实际执行的指令，这使得简单解决方案对模糊测试反馈无效。与最先进的覆盖率引导模糊测试器类似，我们在目标程序的基本块级别上进行工作。此外，我们从目标程序中提取一个**控制流图 (CFG)**，它将基本块表示为节点，并将它们之间可能的转换描述为边。正如我们在本节剩余部分展示的那样，这使我们能够推导出 CFG 的支配关系，这有助于我们减少模糊测试期间的断点中断次数并避免不必要的开销。

#### 1.控制流图的支配关系

**支配关系** 描述了控制流图中节点之间进一步的一致性。我们使用 Agrawal 的前置支配者和后置支配者的概念，并假设控制流图 \(G\) 恰好有一个入口点和一个出口点。

**定义 1（前置支配）。** 如果从入口节点到 \(v\) 的每条路径都包含 \(u\)，则节点 \(u \in G\) 前置支配另一个节点 \(v \in G\)，记作 \(u \stackrel {pre}{\longrightarrow} v\)。

**定义 2（后置支配）。** 如果从 \(v\) 到出口节点的每条路径都包含 \(w\)，则节点 \(w \in G\) 后置支配另一个节点 \(v \in G\)，记作 \(w \stackrel {post}{\longrightarrow} v\)。

支配关系可以表示为（支配者）树，并且可以高效地针对函数进行计算。后置支配树等于反转控制流图的前置支配树。从任何前置和后置支配树中，我们可以推导出关于其他节点的以下传递性知识：

**定理 1（可达性）。** 如果节点 \(v\) 被到达，则前置支配树中的所有父节点之前已被到达，而后置支配树中的所有父节点之后将被到达。

#### 2.过程间控制流图与支配关系

**过程间控制流图**描述整个程序内基本块之间可能的转换，而不仅仅是函数内部。原则上，我们可以通过将所有调用和返回指令添加为边来连接程序中每个函数的控制流图（局部 CFG）。然而，当一个函数有多个调用者时，这会引入歧义，因为它创建了从每个调用函数到每个返回点的路径。在遍历生成的过程间控制流图以计算支配图时，返回边必须仅指向当前函数的实际调用点，这需要**上下文敏感算法**。已发布的上下文敏感算法很复杂，并且实现很少可用。GDBFuzz 仅将支配关系作为减少开销的额外优势使用，因此我们开发了以下简单方法。

我们构建一个**半过程间控制流图**，其中我们通过将所有调用作为从调用点到被调用者的边来连接函数控制流图。在构建半过程间 CFG 时省略返回边，因此不会引入不正确的流。

对于反转的半过程间控制流图，我们反转局部控制流图，跳过调用边，只添加返回边。同样，我们通过移除上下文敏感的调用边来避免插入歧义。然后，可以使用局部控制流图的算法高效计算相应的半过程间支配树。与完整的过程间支配图相比，在最坏情况下，我们可能错过每个调用边的一个支配关系，但这不应真正影响模糊测试性能。

为了方便起见，我们合并前置和后置支配图：\(\{(u,v) | u \stackrel {pre}{\longrightarrow} v \vee u \stackrel {post}{\longrightarrow} v\}\)，这要求我们只处理整个目标程序的单个支配图。

### 四、嵌入式系统模糊测试的挑战

对嵌入式系统进行模糊测试很困难。它们的核心组件 "微控制器" 高度多样化且不易虚拟化；其软件可能无法更改或插桩。然而，我们观察到许多（即使不是大多数）微控制器都配备了一个调试接口，调试探针（通常可通过 GNU 调试器 GDB 控制）可通过该接口设置有限数量的硬件断点。利用这些断点，我们即使对于非插桩的二进制代码也能提取部分覆盖率反馈；从而通过一种通用的、广泛存在的机制实现对嵌入式系统的有效模糊测试。

**1.AFL的局限性**

大多数现代模糊测试器遵循 AFL 的模糊测试模型，然而，将此类模糊测试器应用于嵌入式系统非常困难。原因之一是微处理器、架构和操作系统的高度多样性。但最重要的是，微控制器板卡上的软件不易更改，从而无法进行插桩。

**2.覆盖率收集的困难**

即使可以进行插桩，板卡也需要提供存储空间来捕获覆盖率和其他运行时信息；最后，这些信息需要通过某种硬件接口传回给模糊测试器。因此，在硬件上设置像 AFL 这样的常见模糊测试器需要为每个板卡实现独立的覆盖率收集方案。

**3.虚拟化的缺陷**

近期发表的嵌入式模糊测试方法中很大一部分对嵌入式系统进行虚拟化。然而，这种虚拟化需要在速度和保真度之间进行权衡。更糟糕的是，它不仅需要对微处理器本身进行虚拟化，还需要对板卡上所有其他硬件组件以及它们之间的通信方式进行虚拟化。

## GDBfuzz的实现

### 一、组件

GDBFuzz 由以下组件组成：

**测试数据生成器**：与所有覆盖率引导的模糊测试器一样，GDBFuzz 在输入语料库中保存触发不同代码区域的输入，并通过变异这些输入来衍生新的输入。近年来，已经发布了数十种通用的基于变异的模糊测试器 [34]。因此，我们不是从头开始开发变异算法，而是重用 libFuzzer [32] 的变异引擎。GDBFuzz 中的实际变异引擎很容易替换。

**GDB 控制器**：GDB 控制器管理与 SuT 的调试连接。常见的调试探针通常通过 TCP 套接字提供 GDB 服务器。我们使用 Python 包 `python-gdb-mi` 来发送和接收调试命令，例如设置断点或继续执行。

**目标连接**：目标连接组件是向目标设备发送测试输入的抽象层。它根据实际接口处理连接或断开连接事件。它还处理来自协议的错误反馈。嵌入式系统可能具有大量不同的输入接口和通道，用于接收不受信任的输入。流行的接口包括 Wi-Fi、蓝牙、NFC，也包括各种面向外部的总线，如 CAN、USB、Profibus 或 \(I^{2}C\)。GDBFuzz 可以包含自定义接口适配器以实现广泛的适用性。对于我们的案例研究，我们实现了 TCP、串行和 USB 连接的适配器，以及 UNIX 管道，以便能够对 Linux 应用程序进行模糊测试。

**Ghidra 控制器**：我们使用逆向工程工具 Ghidra 获取目标应用程序的 CFG。为了在 Ghidra 和 GDBFuzz 之间交换请求和数据，我们使用 `ghidra-bridge` Python 包。GDBFuzz 可以连接到正在运行的 Ghidra 实例，或者在目标二进制文件上启动一个无头（headless）实例。

**动态控制流图改进**：如前所述，逆向工程工具无法保证检测到程序的整个控制流 [45]。缺失的控制流表现为 CFG 中的一个孤立节点，没有后继节点，且未被 Ghidra 标记为终端。当找到触发执行此类孤立节点的测试输入时，我们执行以下步骤：

(1) 在孤立节点设置断点，并将触发它的测试输入发送到 SuT。
(2) 中断发生时，执行单步操作。
(3) 读取程序计数器的值。
(4) 将找到的边报告给 Ghidra 并重新分析二进制文件。然后，Ghidra 通常能够基于报告的边恢复更多的控制流。

### 二、覆盖率的收集

本文中，我们提出了一种对现有嵌入式系统进行模糊测试的替代方法，无需虚拟化，而是采用一种适用于各种嵌入式系统的统一方法。大多数微控制器都包含调试单元，调试探针可通过该单元设置断点，执行程序直到断点，并检查当前程序状态，包括程序计数器和内存值。硬件断点是调试单元中的专用寄存器，当程序计数器等于寄存器值时暂停执行，并且即使代码是只读的也可以设置；它们既不修改也不减慢程序代码的速度。

![](../images/learning/2023_issta_gdbfuzz/gdbfuzz.png)

通过基于程序控制流图（CFG）在代码中系统性地设置断点，并检查哪些输入触发了哪些断点，我们可以获取覆盖率信息，从而为反馈驱动的模糊测试策略提供必要的引导。由于微控制器内部的硬件断点数量有限，我们仅将其设置在程序代码块的一个子集上，并定期重定位它们。由于许多调试探针可以通过 GNU 调试器 (GDB) 寻址，我们将上述策略实现为一个名为 **GDBFuzz** 的模糊测试器中，该测试器可以利用任何系统中的 GDB 接口，在覆盖率引导下系统性地生成测试输入。所需的设置如上图所示。

### 三、提取控制流图

在程序编译期间可以轻松获得控制流图，因为编译器了解整个控制流。然而，GDBFuzz 被设计为在二进制文件上工作以扩大其适用性，因为嵌入式系统上可能并非所有软件组件都可用源代码。Ghidra [42] 是一个开源的逆向工程工具，支持最常见的处理器架构，可编写脚本，因此非常适合我们的需求。与所有二进制反汇编方法一样，Ghidra 无法保证检测到所有控制流，特别是在涉及间接分支或激进编译器优化时 [45]。因此，我们在模糊测试期间迭代地改进和更新控制流图，即第一节 "动态控制流图改进" 组件。

### 四、寻找入口点

我们将模糊测试重点放在固件中我们目标输入接口进行输入处理的区域。因此，提取的 CFG 应从输入处理的开始处开始，称为**入口点**。选择入口点是测试工程师的任务，因此他需要了解目标。然而，以下半自动化的寻找合适入口点的方法在我们的分析中被证明是有用的。

(1) 向目标设备发送测试输入并立即中断执行。
(2) 使用 `gdb find` 在设备内存中重新发现发送的输入。
(3) 将数据观察点设置在重新发现的输入数据的第一个地址上。
(4) 再次发送测试输入。
(5) 现在发生中断时的所有程序计数器地址都是入口点的候选者。

这些步骤作为 GDBFuzz 设置的一部分执行一次。

如果二进制文件中包含符号名称，GDBFuzz 也可以使用符号名称作为入口点，以避免每次重新编译后都需要搜索入口点。这在持续集成设置中特别有用，这样新软件版本可以无缝地进行模糊测试。

### 五、工作流程

下图概括了 GDBFuzz 的工作流程。基于目标程序的控制流图（CFG），GDBFuzz 将可用的硬件断点设置在 CFG 中尚未到达的、随机选择的节点上。然后，GDBFuzz 重复生成输入，将其发送到目标设备，并检查它是否触发了表示新代码覆盖率的断点，或者是否使目标系统崩溃。

![](../images/learning/2023_issta_gdbfuzz/process.png)

在我们的实验中，GDBFuzz 被证明可以轻松应用于多个微控制器板卡甚至常规用户应用程序。它比黑盒模糊测试和基于虚拟化的解决方案实现了更高的覆盖率，并且还检测到了一些已知和新的漏洞。总而言之，据我们所知，GDBFuzz 是首个基于硬件的、与架构无关的、独立于源代码的、非侵入性的、易于应用的嵌入式系统覆盖率引导模糊测试方法，我们很高兴将其推荐给任何想要系统测试嵌入式系统鲁棒性的人。

本文其余部分组织如下。第 2 节讨论相关背景。第 3 节解释 CFG 算法和二进制文件的 CFG 提取。第 4 节介绍 GDBFuzz 的设计；第 5 节描述实现细节。我们在第 6 节评估我们的工作，并在第 7 节进一步讨论结果。第 8 节以结论和未来工作结束。GDBFuzz 已作为开源软件提供。


如图 2 所示，GDBFuzz 利用目标程序的控制流图，将可用的硬件断点设置在尚未到达的、随机选择的基本块上。然后，它通过将变异应用于从语料库中随机选择的输入来重复生成测试用例，并将测试用例发送到目标输入接口。如果调试探针发出断点命中信号，GDBFuzz 将相应节点及其支配节点标记为已到达，并将负责的测试用例添加到语料库中。导致崩溃或超时的测试用例被单独保存。当在执行了预定数量的测试用例后没有发生断点中断时，GDBFuzz 将硬件断点重定位到新选择的节点。每次重定位后，GDBFuzz 首先再次测试语料库中的所有输入，以检查它们是否已经到达新定位的基本块。与具有完整代码插桩的覆盖率引导模糊测试一样，进化算法使得输入语料库随着时间增长，包含到达不同代码区域的输入。

本节其余部分描述了我们如何提取 CFG、如何在目标应用程序中找到模糊测试入口点，以及如何在执行期间检测和处理漏洞。

### 六、漏洞的检测、处理

**1.漏洞的检测**

漏洞检测器（Bug Oracle）检测在执行期间是否触发了漏洞。由于模糊测试起源于测试用户应用程序，一个常见的漏洞检测器是观察目标进程是否引发错误信号，例如段错误。为了发现不直接触发错误的漏洞，可以使用消毒器（sanitizer）和断言。这些通常在编译时部署，但也有方法可以直接将消毒器注入二进制文件 [11]。然而，更复杂的漏洞检测器仍然是一个开放的研究问题 [5]，不在本工作范围内。

GDBFuzz 依赖于被触发的漏洞是**可观察的**，这意味着错误或其他异常行为必须由漏洞触发。因此，[39] 中演示的静默损坏（silent corruption）无法被发现，除非在编译期间使用了额外的消毒器。可以通过发生的连接错误（如超时或错误响应代码）来检测错误。此外，可以通过调试接口在错误处理程序的位置设置断点，这也可以捕获已部署消毒器发出的错误信号。

由于代码中错误处理程序的位置在运行时通常不会改变，因此可以使用软件断点来检测它们的执行，这样所有硬件断点都可用于覆盖率反馈机制。在这种情况下，软件断点非常合适，因为它们在模糊测试期间不会被重新定位。如果被测系统 (SuT) 不支持软件断点，也可以使用可用硬件断点的一个子集，但缺点是会降低模糊测试性能。

获取错误处理程序的位置是通过手动到半自动的方式完成的，因为嵌入式系统在处理器、操作系统、框架、库和消毒器等特性上差异巨大。根据我们的经验，例如来自 freeRTOS [20]、Arduino [3] 和 STM32CubeMX [53] 的默认错误处理程序通常以无限循环结束。Ghidra 可以识别具有无限循环的函数 [43]，然后这些函数可以被视为潜在的错误处理程序。对于我们所有的测试应用程序，依靠错误处理程序中无限循环引起的超时就足够了，不需要我们进行任何进一步的设置工作。

**2.漏洞的处理**

每当 GDBFuzz 检测到崩溃或超时时，它都会：

(1) 对漏洞进行去重，以确定该漏洞是否是唯一的，即是否是首次发现此漏洞；
(2) 如果漏洞是唯一的，则保存触发此漏洞的输入；
(3) 重启目标系统；以及
(4) 继续模糊测试。

同一漏洞可能在模糊测试期间被多次触发。分析每个漏洞需要大量工作，因此需要进行去重。目标是为测试工程师提供一个触发唯一漏洞的最小输入集。我们使用调用栈的哈希值 [34] 来唯一标识和去重漏洞。

当触发漏洞时，目标系统可能处于不可恢复状态。同样，如果发生超时，目标系统可能会永远挂起。因此，在发现错误后，我们通过 GDB 重置目标系统。

## 评估

在本节中，我们在两种不同的设置下评估 GDBFuzz，由八个研究问题 (RQs) 引导。

(1) **基于硬件的设置**：我们选择了多种常见的开发板，列在表 1 中，包括相应的架构、使用的调试探针和可用硬件断点的数量。在每个开发板上，我们部署了四类不同的应用程序，列在表 2 中，并提供了初始种子。Buggy 程序将清单 1 中的错误函数暴露给串行输入接口，并作为基准真值。每块板的具体应用程序来源于开发板附带的示例或兼容的工具链。HTTP 和 USB 应用程序类别要求目标板上存在相应的接口。
(2) **基于应用程序的设置**：包含来自 Google 的 Fuzzer Test Suite [24] 的 16 个程序，编译为 x86 Linux 应用程序，提供了一个可扩展且可独立测量的评估环境。GDBFuzz 可以在 QEMU 实例中执行应用程序（支持实时测量到达的代码覆盖率），也可以直接与 GDB 一起执行（支持低开销和无限数量的断点）。我们使用编译器优化 (-O3) 编译应用程序，并在配备四个 Intel Xeon Gold 6144 CPU 和 1.48TB RAM 的服务器上执行相应的实验。

### 一、GDBFuzz 对比黑盒模糊测试

GDBFuzz 使得在难以测量覆盖率的系统上进行覆盖率引导模糊测试成为可能。因此，我们利用 GDBFuzz 自身的部分覆盖率提取机制，在基于硬件的设置中测量 GDBFuzz 与黑盒模糊测试之间的覆盖率差异。具体来说，我们部署 GDBFuzz，但在模糊测试期间省略将新输入添加到语料库，以模拟黑盒模糊测试器。结果，我们可以研究 GDBFuzz 的进化模糊测试算法表现如何，从而解决我们的第一个研究问题：

**RQ1：在嵌入式系统上，GDBFuzz 与黑盒模糊测试相比如何？**

图 3 显示了所有开发板和应用程序类别组合的覆盖率随时间变化的图。每个实验重复两次，累计实验时间为 56 天。在所有运行中，毫无例外，GDBFuzz 都比黑盒模糊测试实现了更高的代码覆盖率，并且表明它可以从通过硬件断点获取的部分覆盖率信息中大大受益。特别是对于 Buggy 程序，如第 2.1 节理论上所述，黑盒模糊测试几乎没有机会满足触发所含栈溢出漏洞的所有条件。在这个应用程序类别中，GDBFuzz 在性能强大的 CY8CKIT 板上每秒可实现近 100 次迭代，而在低性能的 MSP430 板上只能达到每秒约 1.5 次迭代。这解释了为什么 GDBFuzz 在后者上解决输入约束所需的时间要长得多，我们也看到了吞吐量对于模糊测试有多么重要。尽管如此，GDBFuzz 在我们所有的开发板上都表现良好，在所有情况下都找到了漏洞，并正确地报告了由此产生的崩溃。

### 二、GDBFuzz 对比最先进技术

如第2节所述，有多种方法声称能够实现嵌入式系统的覆盖率引导模糊测试，从而引出以下问题：

**RQ2：GDBFuzz 与现有的嵌入式模糊测试方法相比如何？**

\(\mu\)AFL [31] 是一种基于硬件的嵌入式模糊测试器，使用 ARM ETM 接口从嵌入式程序中提取代码覆盖率。我们的 CY8CKIT-062-WIFI-BT 开发板配备了这种 ETM 接口，并且我们拥有所需的跟踪硬件。然而，\(\mu\)AFL 的公开版本在我们的设置上报告了不可信的结果，尽管有供应商支持，我们也未能解决。我们注意到，\(\mu\)AFL 使用了原始跟踪数据函数，这些函数在某些实现中不可靠，并且被标记为仅供内部使用 [22]。此外，与 GDBFuzz 相比，\(\mu\)AFL 需要非常特定的硬件，因此它不是直接竞争对手；我们不知道有通用的基于硬件的嵌入式模糊测试方法可以与 GDBFuzz 进行比较。

大多数已发布的嵌入式模糊测试方法都是基于模拟的，其中只有外设建模方法可以在规模上实现嵌入式系统的覆盖率引导模糊测试，并且是 GDBFuzz 的竞争对手。因此，我们将 GDBFuzz 与最新的外设建模方法 Fuzzware [46] 进行比较，该方法的作者声称在实际硬件上进行嵌入式模糊测试不切实际。Fuzzware 适用于所有基于 ARM Cortex-M 的微控制器，因此我们可以对表 1 中前两个开发板的所有应用程序进行模糊测试。

首先，我们需要商定如何比较基于模拟的方法和基于硬件的方法。Li 等人 [31] 通过每小时实现的模糊测试迭代次数将外设建模方法与他们的基于硬件的方法 \(\mu\)AFL 进行比较。我们同意单位时间内的执行次数是模糊测试的一个重要指标。然而，为了公平比较，在相同或至少相似的时间内必须执行相同或相似的代码区域。像 \(\mathrm{P^2IM}\) [18] 和 Fuzzware [46] 这样的外设建模方法使用模糊测试迭代地为固件构建人工执行环境。随着时间的推移，外设模型被细化，执行速度会下降，因为固件可以进一步执行。从设计上讲，外设建模并不针对特定的代码区域。这使得吞吐量成为比较这些不同方法的无意义指标，因为不清楚在这段时间内是否执行了相同的代码部分。

因此，我们基于模糊测试期间固件目标区域内到达的基本块数量来比较嵌入式模糊测试方法，如 [46] 中所做。

### 三、用 GDBFuzz 揭示漏洞

模糊测试的主要目标是发现软件漏洞，从而引出问题：

**RQ3：该方法能否揭示嵌入式软件代码中的实际漏洞？**

为了回答 RQ3，我们首先看两个已知的真实世界漏洞。STM32CubeL4 USB 中间件 [51] 的 USB 枚举处理包含已知漏洞 CVE-2021-34259 和 CVE-2021-34268，这些漏洞是使用 \(\mu\)AFL [31] 发现的。我们验证 GDBFuzz 可以检测此类真实世界漏洞，并使用基于 STM32CubeL4 [52] 版本 1.17.1 的 USB 主机海量存储设备类 (MSC) 示例应用程序的固件。在此 MSC 应用程序中，STM32 微控制器充当 USB 主机。因此，我们的设置类似于 \(\mu\)AFL 作者的设置。为了生成 USB 流量，我们将一个普通的 USB 闪存驱动器作为 USB 客户端插入 USB 端口。然后，在 USB 枚举的特定阶段引入一个模糊测试工具（fuzzing harness），在 USB 主机处理数据之前，用模糊测试数据替换 USB 数据帧。具体来说，我们替换客户端设备发送的原始设备描述符或设备配置，然后再对模糊测试数据进行任何解析。引入的模糊测试工具通过串行接口从 GDBFuzz 接收模糊测试数据。上述两个 CVE 在触发时表现为超时，因为 USB 主机中间件被格式错误的设备描述符或设备配置 USB 数据包错误配置。基本上，它们是由于缺少对来自 USB 客户端的不受信任数据的有效性检查引起的。在我们的实验中，使用 GDBFuzz 和适当的模糊测试工具，两个 CVE 都在不到 5 分钟的时间内被触发和检测到。

在评估期间，我们发现了三个先前未知的漏洞，并已报告给相应的供应商：

(1) STM32 USB 设备栈中的一个无限循环，由在 for 循环中将一个 uint8_t 索引变量计数到一个攻击者可控的 uint32_t 变量引起 [13]。
(2) Cypress JSON 解析器中的一个缓冲区溢出，由对固定大小的内部缓冲区缺少长度检查引起 [12]。
(3) Cypress JSON 解析器中的一个空指针解引用，由缺少验证检查引起 [14]。

> **结论：** GDBFuzz 揭示了嵌入式软件中的真实漏洞。

### 四、GDBFuzz 对比 AFL++

基于应用程序的设置允许我们使用 GDBFuzz 对 Linux 应用程序进行模糊测试，从而引出研究问题：

**RQ4：GDBFuzz 与最先进的模糊测试器 AFL++ 相比如何？**

为了在 GDBFuzz 和 AFL++ [19] 之间进行公平比较，我们让它们在未插桩的二进制文件上操作，\(\mathrm{AFL++}\) 使用 QEMU 模式 (-Q)，GDBFuzz 也使用 QEMU。由于包含在 \(\mathrm{AFL++}\) 中的修改版 QEMU 无法进行实时测量，我们在模糊测试后重放各自的输入语料库，并测量到达的基本块数量。我们将 GDBFuzz 配置为使用八个断点，这是真实微控制器中可用断点数量较低但现实的情况。

图 4 显示了 GDBFuzz 和 AFL++ 的覆盖率随时间变化的图。显然，\(\mathrm{AFL++}\) 随时间覆盖了比 GDBFuzz 更多的代码。\(\mathrm{AFL++}\) 正是为此类应用程序设计和优化的，可以受益于其详尽的代码插桩。然而，我们认为仅使用八个断点的 GDBFuzz 相差并不太远。在某些应用程序上，GDBFuzz 甚至可以达到相似数量的基本块。我们还强调，\(\mathrm{AFL++}\) 在模拟和插桩不可用的场景下会回退到黑盒模糊测试——而这又是 GDBFuzz 优越的地方。

> **结论：** 如果能以低成本部署 AFL，请使用它；否则，考虑将 GDBFuzz 作为一个潜在要求较低的替代方案。

### 五、支配关系的提升

现在让我们评估 GDBFuzz 设计的特定元素。如第 4 节所述，GDBFuzz 利用支配关系，通过一次断点中断将多个基本块标记为已到达，从而引出问题：

**RQ5：GDBFuzz 从支配关系中受益多少？**

为了回答 RQ5，我们分析了之前实验中（表 4）的平均断点中断次数，以及到达的平均基本块数量。在我们所有的实验中，每次断点中断平均导致标记 3.15 个基本块，这意味着探测的基本块数量减少了 \(68.25\%\)。这个比率比 [56] 中提出的高效代码插桩算法实验中的结果要好，作者在他们的实验中仅将插桩点数量减少了 \(34\%\) 到 \(49\%\)。GDBFuzz 可能能够进一步减少开销，因为我们还使用了后置支配关系和描述的半过程间 CFG。

正如我们在精度列中看到的，绝大多数标记的支配基本块是正确的。错误标记的基本块可能源于逆向工程控制流不正确。主要超过 \(99\%\) 的达到精度对于覆盖率引导模糊测试来说足够了，因为它是一个随机过程，并不依赖于 \(100\%\) 正确的覆盖率数据。流行的模糊测试工具，如 AFL\(^{++}\)，将覆盖率数据存储在哈希映射中，并且由于哈希冲突也会在模糊测试期间遗漏一些覆盖率。

> **结论：** 在我们所有的实验中，支配关系将所需的断点中断减少了三分之二以上。

如第 5 节所述，GDBFuzz 可以引导逆向工程工具揭示未被检测到的控制流，我们通过以下问题进行研究：

**RQ6：GDBFuzz 能在多大程度上帮助逆向工程工具揭示未被识别的控制流？**

我们通过比较平均相对数量的额外揭示的基本块和边与 Ghidra 初始检测到的数量来回答 RQ6。由于目标在激活编译器优化的情况下编译，对于逆向工程工具来说，恢复控制流尤其困难。在表 4 中，我们可以看到，在我们的实验中，最多可以揭示 \(674.16\%\) 的额外基本块和 \(708.61\%\) 的额外边。新发现控制流的数量较少并不一定表明 GDBFuzz 的性能较低，而是表明 Ghidra 的逆向工程性能良好。

> **结论：** GDBFuzz 在模糊测试期间为逆向工程揭示了未被检测到的基本块和边。

可用断点的数量在不同的微控制器系列和型号之间各不相同，从而引出问题：

**RQ7：可用断点的数量如何影响模糊测试性能？**

为了评估不同数量的断点如何影响模糊测试性能，我们直接用 GDB 执行应用程序，并使用普通的软件断点进行反馈，因为 QEMU 在断点数量增加时扩展性不好。这样我们就有任意多的断点可用，而不会影响执行时间，并且可以估计它们的数量如何影响随时间达到的覆盖率。为了回答 RQ7，我们在基于应用程序的设置中执行 GDBFuzz，使用从 1 到 65536 呈指数增长的虚拟断点数量。作为代表，图 5 显示了四个应用程序上随时间达到的基本块，来自 2 次运行的平均值。

不出所料，使用更多的断点会导致单位时间内覆盖更多的代码块。在我们的实验中，大致看来，将断点数量翻倍会导致模糊测试性能的线性提升。努力与收益之间的指数相关性在模糊测试研究领域很常见 [6]。同样，我们关于所用断点数量与随时间覆盖率之间的实验观察也表明了指数相关性。

> **结论：** 单位时间内获得线性的更多覆盖率需要指数级更多的断点。

如第 6.4 节所述，当没有符合要求的插桩机制时，AFL\(^{++}\) 会回退到黑盒模糊测试。这引出了我们的最后一个研究问题：

**RQ8：在基于应用程序的设置中，GDBFuzz 与黑盒模糊测试相比如何？**

为了比较 GDBFuzz 和黑盒模糊测试，我们在模糊测试期间使用 QEMU 直接测量到达的基本块，因为黑盒模糊测试没有语料库可以重放。

图 6 显示了黑盒模糊测试、GDBFuzz 以及一个简化版本（GDBFuzzSimple，不使用第 3 节描述的支配关系）的覆盖率随时间测量的结果。这个具有独立代码覆盖率测量的大规模基准测试证实了来自开发板的结果：在所有实验中，GDBFuzz 都优于黑盒模糊测试。此外，我们可以看到，使用支配关系来获得传递性知识在大多数目标应用程序上导致了更多和更快的代码覆盖率。尽管如此，即使是 GDBFuzzSimple 也在所有实验中大大优于黑盒模糊测试。

## 讨论

既然我们已经展示了 GDBFuzz 的良好效果，我们想讨论使用代码块覆盖率作为指标的设计选择、使用 GDBFuzz 进行模糊测试的实践方式，以及我们评估实验的有效性威胁。

### 一、块覆盖率和边覆盖率

虽然大多数最先进的模糊测试器（如 AFL++ [19] 和 libFuzzer [32]）利用边覆盖率，但我们使用块覆盖率来引导进化模糊测试算法。用我们的方法提取边覆盖率将需要多次探测已到达的块，这会急剧增加开销。Nagy 等人 [40] 使用软件断点来检测普通软件中新基本块的执行。他们发现，由于所需的恒定插桩开销，边覆盖率无法从其更细的粒度中受益。由于在我们的设置中，边插桩将比普通软件的源代码插桩引入更多的开销，我们估计块覆盖率是 GDBFuzz 中唯一可行的覆盖率指标。

### 二、模糊测试固件驱动程序

模糊测试驱动程序或嵌入式软件的中间件可以通过黑盒方法实现，即通过将模糊测试数据注入面向外部的接口；或通过白盒方法实现，即将模糊测试工具编译到固件中，以获取模糊测试数据并将其重定向到驱动程序函数。白盒方法提供更大的灵活性，因为可以直接调用驱动程序函数，并且可以向模糊测试器反馈确认信号，表明函数已正确返回。然而，需要关于代码和系统的专业知识来实现合适的模糊测试工具。此外，实现的工具仅适用于特定的代码库，并且可能产生误报，因为在到达被测试的驱动程序函数之前，输入可能已在硬件中被清理 [31]。白盒方法适用于复现 \(\mu\)AFL 揭示的已知 CVE，因为错误的函数是已知的，并且原始的发现者也选择了同样的方式。

黑盒方法通常需要较少的设置工作，因为在大多数测试设置中，通往目标接口的数据路径应该已经存在，因此测试工程师只需要通过实现合适的 Python 类将现有路径连接到 GDBFuzz。因此，我们的连接适配器可以开箱即用，例如，适用于所有现有的 USB 设备驱动程序。我们使用 GDBFuzz 发现的先前未知的漏洞，是在未经更改的固件上触发的，无需开发和使用复杂的工具函数。
